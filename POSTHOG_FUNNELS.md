# Настройка воронок PostHog для Deliverty

Этот документ описывает, как настроить воронки (funnels) в PostHog для анализа пользовательского поведения в приложении Deliverty.

## Полный список отслеживаемых событий

Приложение отслеживает следующие события для построения воронок:

### Общие события
- `$pageview` - просмотр любой страницы (автоматически отслеживается PostHog)
- `navigation_clicked` - клик по навигации в шапке (свойства: `destination`, `from`)
- `bottom_nav_clicked` - клик по нижней навигации на мобильных (свойства: `destination`, `from`)

### Авторизация
- `auth_page_viewed` - открытие страницы авторизации (свойства: `return_url`)
- `auth_success` - успешная авторизация через Telegram (свойства: `user_id`, `return_url`)
- `auth_error` - ошибка авторизации (свойства: `error`)
- `telegram_bot_link_clicked` - клик на ссылку бота в Telegram (свойства: `source`)

### Создание публикации
- `publish_page_viewed` - открытие страницы создания публикации (свойства: `kind`, `from_url_param`, `to_url_param`)
- `publish_kind_changed` - изменение типа публикации (свойства: `kind`: "request" | "trip")
- `publish_started` - начало процесса публикации (нажатие кнопки "Опубликовать") (свойства: `kind`, `from_iata`, `to_iata`, `item`, `weight`, `has_description`)
- `publish_completed` - успешное создание публикации (свойства: `pub_id`, `kind`, `from_iata`, `to_iata`, `item`, `weight`, `has_description`)
- `publish_error` - ошибка при создании публикации (свойства: `kind`, `reason` или `error`, `from_iata`, `to_iata`)
- `publish_attempted_not_authenticated` - попытка публикации без авторизации (свойства: `kind`)
- `create_publication_from_profile` - клик на создание публикации со страницы профиля

### Поиск и просмотр
- `browse_page_viewed` - открытие страницы поиска
- `filter_changed` - изменение фильтра поиска (ищу/лечу) (свойства: `filter`, `from_iata`, `to_iata`)
- `airport_selected` - выбор аэропорта в поле ввода (свойства: `iata`, `airport_name`, `field_label`)
- `search_started` - начало поиска (свойства: `from_iata`, `to_iata`, `kind_filter`)
- `search_completed` - завершение поиска с результатами (свойства: `from_iata`, `to_iata`, `kind_filter`, `results_count`)
- `search_results_viewed` - просмотр результатов поиска (только если есть результаты) (свойства: `from_iata`, `to_iata`, `kind_filter`, `results_count`)
- `search_error` - ошибка при поиске (свойства: `reason` или `from_iata`, `to_iata`, `kind_filter`, `error`)
- `search_cleared` - очистка результатов поиска (свойства: `from_iata`, `to_iata`, `kind_filter`)
- `navigate_to_publish` - переход на страницу создания публикации из поиска (свойства: `from_search`, `kind`, `from_iata`, `to_iata`)
- `request_contacts_clicked` - клик на кнопку "Показать контакты" (свойства: `pub_id`, `pub_kind`, `from_iata`, `to_iata`)
- `telegram_link_clicked` - клик на ссылку Telegram (свойства: `link`)

### Сделки
- `deal_started` - начало диалога между пользователями (бекенд отправляет при создании `contact_request`)
- `deal_created` - подтверждение сделки (бекенд отправляет из обработчика Telegram-кнопки «✅ Договорились»)
- `deal_error` - отказ от сделки (бекенд отправляет из кнопки «❌ Не договорились»)
- `deal_attempted_not_authenticated` - зарезервировано (фронт пока не задействует `makeDeal`)

> 💡 События `deal_*` теперь приходят из бекенда, поэтому их можно использовать в воронках вместе с фронтовыми шагами (`request_contacts_clicked` и т.д.).

### Профиль
- `profile_page_viewed` - открытие страницы профиля (свойства: `publications_count`)
- `publication_status_toggled` - изменение статуса публикации (активно/неактивно) (свойства: `pub_id`, `new_status`)
- `publication_updated` - успешное обновление публикации (свойства: `pub_id`, `is_active`)
- `publication_update_error` - ошибка при обновлении публикации (свойства: `pub_id`, `error`)
- `profile_error` - ошибка при загрузке профиля (свойства: `error`)

## Примеры воронок

### 1. Воронка создания публикации (базовая)

**Цель:** Проанализировать, сколько пользователей доходят до успешного создания публикации.

**События в порядке:**
1. `publish_page_viewed` - открытие страницы
2. `publish_started` - начало публикации
3. `publish_completed` - успешное создание

**Расширенная воронка с заполнением формы:**
1. `publish_page_viewed`
2. `airport_selected` (где `field_label = "Откуда"`) - выбор аэропорта отправления
3. `airport_selected` (где `field_label = "Куда"`) - выбор аэропорта назначения
4. `publish_started`
5. `publish_completed`

**Как настроить в PostHog:**
1. Зайдите в PostHog → Insights → New Insight
2. Выберите "Funnel"
3. Добавьте события в указанном порядке
4. Настройте временной интервал (например, 1 час или 1 день)

**Метрики для анализа:**
- Конверсия на каждом шаге
- Где теряются пользователи
- Время между шагами
- Процент пользователей, заполняющих оба аэропорта

### 2. Воронка поиска и сделки

**Цель:** Понять, сколько пользователей находят совпадения и создают сделки.

**События в порядке:**
1. `browse_page_viewed` - открытие страницы поиска
2. `airport_selected` (где `field_label = "Откуда"`) - выбор аэропорта отправления
3. `airport_selected` (где `field_label = "Куда"`) - выбор аэропорта назначения
4. `search_started` - начало поиска
5. `search_completed` где `results_count > 0` - успешный поиск с результатами
6. `search_results_viewed` - просмотр результатов
7. `request_contacts_clicked` - запрос контактов
8. `deal_started` - бекенд подтвердил создание `contact_request`
9. `deal_created` - бекенд получил подтверждение сделки в Telegram

**Упрощенная версия:**
1. `browse_page_viewed`
2. `search_completed` где `results_count > 0`
3. `deal_started`
4. `deal_created`

**Вариант с фильтрацией:**
- `search_completed` где `results_count > 0` - только поиски с результатами
- `request_contacts_clicked` с фильтром по `pub_kind` - фронтовые взаимодействия с конкретным типом объявлений
- `deal_started`/`deal_created` с фильтром `pub_kind` или `trigger` (разделение по каналам Web/Telegram)

### 3. Воронка авторизации и первого действия

**Цель:** Понять, сколько новых пользователей авторизуются и выполняют первое действие.

**События в порядке:**
1. `auth_page_viewed` - просмотр страницы авторизации
2. `auth_success` - успешная авторизация
3. `publish_page_viewed` ИЛИ `browse_page_viewed` - переход к использованию
4. `publish_completed` ИЛИ `search_completed` - первое успешное действие

**Фильтры:**
- Можно добавить фильтр по новым пользователям: `$is_first_session = true`
- Можно отследить только тех, кто вернулся после авторизации: `auth_success` где `return_url` не пустой

### 4. Воронка полного цикла (поиск → сделка)

**Цель:** Проследить путь от поиска до подтверждённой сделки.

**События в порядке:**
1. `browse_page_viewed` - открытие поиска
2. `search_completed` где `results_count > 0` - успешный поиск с результатами
3. `search_results_viewed` - просмотр результатов
4. `request_contacts_clicked` - взаимодействие с результатом (фронт)
5. `deal_started` - контакт создан (бекенд)
6. `deal_created` - сделка подтверждена (Telegram)

**Альтернативный путь:**
1. `browse_page_viewed`
2. `search_completed` где `results_count = 0` - поиск без результатов
3. `navigate_to_publish` - переход на создание публикации
4. `publish_completed` - создание публикации

> 📌 Планируемое расширение: если/когда кнопка «Создать сделку» вернётся, добавьте шаги `deal_started` и `deal_created` в эту воронку или создайте отдельную ветку для сделок.

### 5. Воронка создания публикации с разбивкой по типу

**Цель:** Сравнить конверсию для запросов (request) и поездок (trip).

**События:**
1. `publish_page_viewed`
2. `publish_kind_changed` где `kind = 'request'` ИЛИ `kind = 'trip'`
3. `publish_started` где `kind = 'request'` ИЛИ `kind = 'trip'`
4. `publish_completed` где `kind = 'request'` ИЛИ `kind = 'trip'`

**Создайте две отдельные воронки:**
- Одна для `kind = 'request'`
- Другая для `kind = 'trip'`

Или используйте **Breakdown** по свойству `kind` в событии `publish_completed`.

### 6. Воронка заполнения формы публикации

**Цель:** Понять, на каком этапе заполнения формы пользователи отказываются.

**События:**
1. `publish_page_viewed`
2. `publish_kind_changed` - выбор типа публикации
3. `airport_selected` где `field_label = "Откуда"` - выбор аэропорта отправления
4. `airport_selected` где `field_label = "Куда"` - выбор аэропорта назначения
5. `publish_started` - начало отправки формы
6. `publish_completed` - успешная публикация

**Анализ:**
- Если конверсия падает после `publish_kind_changed`, возможно, нужно улучшить выбор типа
- Если падает после выбора первого аэропорта, возможно, поле "Куда" неочевидно
- Если падает после выбора обоих аэропортов, возможно, форма слишком сложная

### 7. Воронка взаимодействия с результатами поиска

**Цель:** Понять, как пользователи взаимодействуют с найденными публикациями.

**События:**
1. `search_results_viewed` - просмотр результатов
2. `request_contacts_clicked` - запрос контактов
3. `deal_started` - создание контакт-запроса
4. `deal_created` - подтверждение сделки (или `deal_error` для отказов)

**Анализ:**
- Процент пользователей, которые запрашивают контакты
- Разница между запросами контактов и реально подтверждёнными сделками

## Настройка воронок в PostHog UI

### Шаг 1: Создание новой воронки

1. Войдите в PostHog
2. Перейдите в **Insights** → **New Insight**
3. Выберите тип **Funnel**

### Шаг 2: Добавление событий

1. Нажмите **Add step**
2. Выберите событие из списка или введите название вручную
3. (Опционально) Добавьте фильтры:
   - По свойствам события (например, `kind = 'request'`)
   - По пользователям (например, новые пользователи: `$is_first_session = true`)
   - По времени (например, последние 7 дней)
   - По URL (например, `$current_url` содержит определенный путь)
4. Повторите для каждого шага воронки

### Шаг 3: Настройка параметров

- **Time window**: Временной интервал для завершения воронки (например, 1 час, 1 день, 7 дней)
- **Conversion window**: Максимальное время между шагами (по умолчанию 14 дней)
- **Date range**: Период анализа данных (например, последние 30 дней)
- **Breakdown**: Разбивка по свойствам (например, по типу публикации `kind`, по фильтру `kind_filter`)
- **Conversion rate**: Выберите, как считать конверсию (по всем пользователям или только по тем, кто дошел до первого шага)

### Шаг 4: Сохранение и отслеживание

1. Нажмите **Save** для сохранения воронки
2. Добавьте название и описание
3. Воронка будет доступна в списке Insights
4. Можно добавить в Dashboard для постоянного мониторинга

## Советы по анализу

### Проблемные места

- **Если конверсия падает на `publish_started` → `publish_completed`**: 
  - Форма может быть слишком сложной
  - Проверьте ошибки валидации через событие `publish_error`
  - Возможно, нужно упростить процесс

- **Если падает на `search_completed` → `deal_started`**: 
  - Результаты могут быть нерелевантными
  - Проверьте, сколько результатов показывается (`results_count`)
  - Возможно, нужно улучшить алгоритм поиска

- **Если падает на `deal_started` → `deal_created`**: 
  - Процесс создания сделки может быть слишком сложным
  - Проверьте ошибки через событие `deal_error`
  - Возможно, нужно автоматизировать создание публикаций

- **Если падает на `airport_selected` → `publish_started`**: 
  - Пользователи могут не понимать, как заполнить форму полностью
  - Проверьте, заполняют ли они оба аэропорта
  - Возможно, нужно добавить подсказки

### Оптимизация

- **Используйте разбивку (breakdown)** по свойствам для выявления паттернов:
  - По типу публикации (`kind`)
  - По фильтру поиска (`kind_filter`)
  - По маршруту (`from_iata`, `to_iata`)
  
- **Сравнивайте воронки по периодам** для оценки улучшений:
  - Создайте копию воронки для сравнения
  - Измените даты и сравните результаты

- **Анализируйте время между шагами** для выявления узких мест:
  - PostHog показывает среднее время между шагами
  - Если время слишком долгое, возможно, процесс слишком сложный

- **Используйте фильтры** для анализа специфических сегментов:
  - Только новые пользователи
  - Только определенные маршруты
  - Только определенный тип публикации

### Примеры полезных метрик

**Конверсия создания публикации:**
```
(publish_completed / publish_page_viewed) * 100
```

**Конверсия заполнения формы:**
```
(airport_selected где field_label="Куда" / airport_selected где field_label="Откуда") * 100
```

**Конверсия сделки:**
```
(deal_created / search_completed where results_count > 0) * 100
```

**Конверсия от просмотра результатов до сделки:**
```
(deal_created / search_results_viewed) * 100
```

**Среднее время от поиска до сделки:**
```
Average time between search_completed and deal_created
```

**Процент пользователей, запрашивающих контакты:**
```
(request_contacts_clicked / search_results_viewed) * 100
```

## Дополнительные возможности

### Когортный анализ
Создайте когорты пользователей по дате первого визита и сравните их воронки:
1. Создайте когорту в PostHog (Users → Cohorts)
2. Используйте когорту как фильтр в воронке
3. Сравните конверсию между когортами

### Retention
Используйте события для анализа retention:
- Пользователи, которые вернулись после `publish_completed`
- Пользователи, которые создали несколько сделок
- Пользователи, которые создали публикацию после поиска

### А/B тестирование
Если вы проводите A/B тесты, используйте свойства событий для разбивки результатов по вариантам:
- Добавьте свойство `experiment_variant` в события
- Используйте Breakdown по этому свойству в воронках

### Сегментация пользователей
Создайте сегменты пользователей для более глубокого анализа:
- Новые пользователи (`$is_first_session = true`)
- Вернувшиеся пользователи
- Пользователи, создавшие публикацию
- Пользователи, создавшие сделку

## Отладка событий

### Проверка отправки событий

1. **В PostHog UI:**
   - Перейдите в **Events** → **Live events**
   - Выполните действие в приложении
   - Проверьте, что событие появилось в реальном времени

2. **В браузере:**
   - Откройте DevTools (F12)
   - Перейдите на вкладку Network
   - Отфильтруйте по "posthog" или "e"
   - Проверьте, что запросы отправляются

3. **В коде:**
   - Убедитесь, что `VITE_PUBLIC_POSTHOG_KEY` установлен
   - Проверьте, что `usePostHogAnalytics` используется правильно

### Типичные проблемы

- **События не появляются:**
  - Проверьте, что PostHog ключ настроен
  - Проверьте, что события отправляются (DevTools → Network)
  - Убедитесь, что названия событий точно совпадают (учитывается регистр)

- **Фильтры не работают:**
  - Проверьте, что свойства событий заполнены правильно
  - Убедитесь, что тип свойства совпадает (строка, число, boolean)
  - Проверьте синтаксис фильтра в PostHog

- **Воронка показывает 0%:**
  - Проверьте временной интервал (может быть слишком коротким)
  - Проверьте, что события происходят в правильном порядке
  - Убедитесь, что фильтры не исключают все события

## Поддержка

Если у вас возникли вопросы по настройке воронок:
1. Проверьте, что события правильно отправляются в PostHog (Events → Live events)
2. Убедитесь, что названия событий точно совпадают (учитывается регистр)
3. Проверьте фильтры - они могут исключать нужные события
4. Проверьте временные интервалы - они могут быть слишком короткими
5. Используйте Debug mode в PostHog для детального анализа
